El flujo

Un push. Cinco stages. Jira actualizado solo.

01
git push feature/mi-cambio
El developer hace push a su rama. Nada más. No abre Jira. No crea PR. No toca nada.
02
Stage 1 — Jira + Build
El pipeline detecta si la rama tiene un SCRUM-NNN. Si lo tiene, toma esa tarjeta. Si no, crea una nueva en el Sprint activo.
Jira: Por hacer → En curso
03
Stage 2 — Pull Request automático
Se crea el PR hacia main automáticamente. El link del PR se comenta en la tarjeta de Jira.
Jira: comentario con link del PR
04
Stage 3 — Merge automático
El PR se mergea a main sin intervención humana. La tarjeta avanza.
Jira: En curso → In Review
05
Stage 4 — Docker → Azure Container Registry
Se construye la imagen Docker y se publica en ACR con tag del commit SHA. Se comenta en Jira.
Jira: comentario con imagen publicada
06
Stage 5 — Deploy → Azure Kubernetes Service
La nueva imagen se despliega en AKS. Se espera el rollout. Si todo está verde, la tarjeta se cierra. Si algo falla, vuelve a Por hacer con el link al log.
Jira: In Review → Listo ✓ (o → Por hacer si falla)
Escenarios de uso

Dos convenciones. Cero fricción.

Escenario A — Tarea existente

git checkout -b feature/SCRUM-3-fix git push origin feature/SCRUM-3-fix

El pipeline detecta SCRUM-3 en el nombre de la rama y mueve esa tarjeta específica por todos los stages.

Escenario B — Tarea nueva

git checkout -b feature/nuevo-endpoint git push origin feature/nuevo-endpoint

No hay SCRUM-NNN. El pipeline crea automáticamente una nueva tarjeta en el Sprint activo y la mueve de principio a fin.

Escenario C — Fallo en producción

Stage 5: kubectl rollout → ERROR → Jira: Listo → Por hacer → Comentario: link al log exacto

Si el deploy falla, la tarjeta regresa a Por hacer automáticamente con el link directo al error en GitHub Actions.

Escenario D — Pipeline de voz (próximo)

"vamos a hacer el MVP 450" → Claude crea SCRUM-450 en Jira → Crea rama feature/SCRUM-450 → Pipeline completo se dispara

Activación por voz. Claude orquesta todo. El developer ni abre el navegador.
